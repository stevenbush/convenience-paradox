# Service Cost 机制审计报告

**日期**: 2026-04-01  
**稳定版 campaign**: `20260401_125557_full_campaign`  
**引擎**: `stable` (`ConvenienceParadoxModel`)  
**目的**: 在解释任何 `research_v2` 新结果之前，先把当前稳定模型里这些“反直觉现象”为什么会出现讲清楚。

## 审计范围

本报告针对你提出的 5 个问题，先只解释**当前稳定模型**：

1. `service_cost_factor` 到底怎样影响 delegation、labor 和 stress？
2. 为什么在当前模型里，`service_cost_factor` 越低，平均总劳动反而越低？
3. 为什么更便利、更偏好委托的基线社会，平均压力反而更低？
4. 为什么当前实验里，过载主因是 task pressure，而不是 delegation 本身？
5. 这些现象更像是参数问题、公式问题，还是扫描范围问题？

本报告只使用现有 `stable` 模型和 `20260401_125557_full_campaign` 输出，不混入任何 `research_v2` 新实验。

## 简短结论

当前结果在稳定模型内部是**结构性可解释**的，不是单纯噪声。

有 3 个稳定版事实几乎解释了全部反直觉现象：

1. `service_cost_factor` 降低，会稳定地提高 realised delegation。
2. 在稳定模型里，delegation 通常会让 requester 节省的时间，大于 provider 额外承担的时间。
3. 供给稀缺几乎没有被真正激活，因为当前 atlas 里 `unmatched_tasks` 基本一直为 0。

因此，在当前稳定模型下，廉价服务更像是一个“减压阀”，而不是一个“恶性放大器”。

而且这不是简单平均造成的错觉。因为在每一个固定 `delegation_preference_mean` 切片里，都同样观察到：

- cost 越高，realised delegation 越低
- cost 越高，总劳动越高
- cost 越高，平均压力越高

也就是说，在当前稳定模型里，“便宜服务伴随更低 labor、更低 stress”并不是偶然，而是稳定出现的。

## 证据 2：为什么低价格会让平均总劳动下降

这其实是当前机制里最核心的一点。

在稳定模型中，provider 执行任务的时间，很多时候比“平均自己做”的时间还短。

当前公式下：

- 自己做的 skill 来自 `0.3` 到 `0.9` 的均匀分布
- provider skill 固定为 `0.6`
- 如果 skill 低于任务要求，会额外吃到明显的时间惩罚

按当前公式计算，期望耗时如下：

| 任务类型 | 平均自做耗时 | provider 在 skill=0.6 时耗时 | provider 相对平均自做 |
| --- | ---: | ---: | ---: |
| domestic | 1.4653 | 1.3333 | -9.0% |
| administrative | 2.4206 | 2.0000 | -17.4% |
| errand | 0.9158 | 0.8333 | -9.0% |
| maintenance | 3.5120 | 2.7500 | -21.7% |
| 总体均值 | 2.0785 | 1.7292 | -16.8% |

这意味着：在当前稳定模型里，delegation 不是 labor-neutral 的，反而通常是 labor-saving 的。

所以就会出现这样的链路：

- 服务更便宜
- 更多任务转去 delegation
- delegation 这条执行路径平均比“自己做”更省时间
- requester 又几乎不承担显著的委托协调时间
- 最终总劳动反而下降

这正是你觉得反直觉的现象在当前模型里会出现的主因。

## 证据 3：为什么低价格还会伴随更低压力

在稳定模型里，stress 只由“当天剩余时间是否跌破阈值”决定，而不是由 delegation 本身决定。

当前 stress 更新里并没有：

- 委托本身带来的直接压力惩罚
- 等待服务带来的直接压力惩罚
- 对速度期待落空的惩罚
- 对依赖服务体系本身的心理成本建模

因此，在当前公式下，便宜服务会沿着最直接的通道降低 stress：

- 服务更便宜 -> delegation 更多
- delegation 更多 -> requester 自己消耗的时间更少
- 剩余时间更不容易跌破 `stress_threshold`
- 低于阈值的人变少 -> 平均 stress 下降

所以，“低价格伴随更低 stress”在当前稳定模型里其实是顺着现有公式自然推出的，并不矛盾。

## 证据 4：为什么供给稀缺在当前稳定模型里几乎没真正发作

你直觉中的恶性循环之所以没出来，一个很重要的原因是：当前模型把服务供给写得太“宽松”了。

有两个关键机制：

1. provider 资格判定很宽松。只要 `available_time >= 0.5 * base_time`，就能接单。
2. 委托失败的任务只会在当步记成 `unmatched_tasks`，不会回流成下一步 backlog。

而在现有数据里，这个宽松机制带来了非常明确的后果：

- 在稳定版 Package C 的 cost atlas 里，所有 cost 水平的 `unmatched_tasks` 平均值都是 `0.0`
- 在稳定版 Package B 的 delegation x task-load atlas 里，即使 task load 到 `3.5`，每个格子的 `tail_unmatched_tasks_mean` 最大值仍然是 `0.0`

这意味着当前模型几乎不会进入你预期中的这个阶段：

- 大家更爱委托
- 供给跟不上
- 任务积压
- 积压又把明天压力抬得更高

因为在稳定模型里：

- delegation 的确会增加 provider 工作
- 但失败任务不会积压到明天
- matching 又几乎总能成功
- 所以系统并不会累积出持续性的服务缺口

这就是为什么你直觉中的“恶性循环”在当前实验里没有真正跑出来。

## 证据 5：为什么便利导向基线会出现更低平均压力

看稳定版 Package A 在 200 steps 时的基线比较：

| 场景 | 尾部平均压力 | 尾部总劳动 | 尾部委托占比 | 最终平均剩余时间 | 最终 p10 剩余时间 | 最终 p90 剩余时间 | 时间 Gini |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| Type A baseline | 0.0346 | 429.4746 | 0.0899 | 3.6915 | 0.9868 | 5.5850 | 0.2578 |
| Type B baseline | 0.0116 | 503.6110 | 0.6324 | 2.9793 | 1.8181 | 3.9169 | 0.1596 |

稳定版 Type B 的特点是：

- 总劳动更高
- 平均剩余时间更低
- 但底部 10% 的剩余时间反而更高
- 时间不平等显著更低

而 stress 又是一个阈值型指标，所以这组结构就足以让平均 stress 更低。

换句话说，当前稳定模型其实在说：

- convenience 会抬高整体劳动总量
- 但它也会把时间压力重新分配得更“均匀”
- 结果是更少的人深度跌破 stress threshold

所以“总劳动更高但平均 stress 更低”在当前稳定模型里是完全可能的。

## 证据 6：为什么当前主导过载机制是 task pressure

稳定版 Package B 给出的 task load 平均结果如下：

| task load | 平均压力 | 总劳动 | 时间不平等 | 委托占比 | 未匹配任务 |
| --- | ---: | ---: | ---: | ---: | ---: |
| 1.5 | 0.0058 | 298.8235 | 0.1422 | 0.3719 | 0.0 |
| 2.0 | 0.0120 | 376.5141 | 0.1716 | 0.3756 | 0.0 |
| 2.5 | 0.0269 | 464.2865 | 0.2021 | 0.3839 | 0.0 |
| 3.0 | 0.0829 | 549.0548 | 0.2352 | 0.4039 | 0.0 |
| 3.5 | 0.7235 | 626.3568 | 0.2638 | 0.5858 | 0.0 |

而且在每个固定 task-load 切片里：

- `delegation_preference_mean` 与 `total_labor` 的相关性都强烈为负
- `delegation_preference_mean` 与 `avg_stress` 的相关性也为负

所以在当前稳定模型里，delegation 在固定负载切片内更像是“缓冲器”，而 task load 的上升才是真正把系统推过 stress threshold 的主因。

因此，“当前模型里的主导过载机制是 task pressure，而不是 delegation 本身”这个结论在稳定模型中是成立的。

但这并不表示 delegation 不重要，而是表示：

- 当前模型缺少足够强的 backlog / scarcity 机制
- 因此 delegation 还没有被建成一个可以持续放大过载的主通道

## 诊断：这是不是模型有问题？

是的，但更准确地说，这是一个**机制覆盖不足**的问题，而不是简单的“参数写错了”。

当前稳定模型最主要的缺口有 5 个：

1. **没有 backlog carryover**  
   委托失败不会变成明天继续压在 requester 身上的任务。

2. **没有 requester coordination cost**  
   委托会花钱，但几乎不花时间。

3. **provider 执行平均过快**  
   当前固定 provider skill=`0.6`，在现有 skill 分布下，比平均自做还快。

4. **matching 容量过宽松**  
   只要有 `0.5 * base_time` 空闲时间就能接单，而不是按完整预期服务时长判断。

5. **stress 只看阈值，不看等待与依赖**  
   没有等待成本、失败委托成本、不确定性成本。

这些缺口正好会压制你预期中的“delegation 诱发恶性循环”。

## 扫描范围是不是也偏窄？

部分是。

稳定版扫描已经能看出 task load 上升会触发明显过载跳变，但它仍然没有触发真正的 delegation-capacity crisis。原因不只是扫描范围，还包括：

- 扫描范围本身
- backlog 动态缺失
- provider 匹配过宽松
- provider 执行过高效

所以这不是单一原因。

## 对你 5 个问题的稳定版回答

1. **`service_cost_factor` 应该怎么理解？**  
   在稳定模型里，它主要影响 delegation uptake，而不是服务容量本身。低 cost 会提高 realised delegation，并在当前机制下伴随更低 labor 和更低 stress。

2. **为什么低 cost 会让平均总劳动更低？**  
   因为 delegation 路径在当前模型里平均比“自己做”更省时间，而且 requester 几乎不承担委托协调时间。

3. **为什么更偏爱委托的社会会有更低平均压力？**  
   因为 stress 由剩余时间阈值决定，而当前 delegation 机制确实帮助更多 agent 留在阈值之上。

4. **为什么当前模型没跑出强烈的 delegation 恶性循环？**  
   因为 backlog 不会累积，matching 几乎总能成功，服务稀缺没法持续传播。

5. **这说明 preset 或公式有问题吗？**  
   更准确地说，是机制覆盖不够、部分公式过宽松，不一定是 preset 初衷本身有问题。

## 下一步

正确的下一步不是去改 dashboard 模型，而是启用一个独立研究引擎，同时保持网页契约不变。这个研究引擎需要补上：

- backlog 回流
- requester 协调时间
- provider 服务摩擦
- 按预期服务时长判断 provider 资格
- 显式 labor 分账与 match-rate 指标
