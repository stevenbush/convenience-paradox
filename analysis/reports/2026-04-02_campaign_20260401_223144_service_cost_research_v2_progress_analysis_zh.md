## 1. 数据基础与实验设计

本组实验是一次聚焦型机制验证，不追求包覆盖最大化，而是集中回答两个问题：

1. Type A / Type B 两类预设在研究引擎下是否仍表现出稳定差异。
2. `service_cost_factor` 到底是在“减压”，还是在某个容量阈值后转而放大积压与压力。

本次可复核的数据基础如下：

- `summaries/combo_summary.csv`：618 个聚合场景
- `summaries/per_seed_summary.csv`：4152 条 seed 级记录
- `summaries/research_metric_probe.csv`：补充的 research-only 指标，包含 backlog、match rate、stress breach、time deficit
- `Package A` 使用 12 seeds 的 horizon scan
- `Package C` 的 context scan 使用 12 seeds，`service_cost_task_load_atlas` 使用 6 seeds

结构上，本组实验包含三块核心证据：

- `Package A / preset_horizon_scan`：6 个情景，比较 Type A 与 Type B 在 120、200、300 步下的尾窗指标
- `Package C / service_cost_context_scan`：80 个情景，比较 4 种上下文下的价格摩擦作用
- `Package C / service_cost_task_load_atlas`：520 个情景，对两档高委托强度下的 `service_cost x task_load` 阈值区域做剖面扫描

## 2. 核心结论

1. 在研究引擎下，Type B 不再表现为“更便利但更低压”，而是持续表现为“更高委托、更高总劳动、可支配时间更少，而且平均压力也更高”。
2. `service_cost_factor` 的作用是强条件性的：在低负载区间，低价服务仍然减压；但当任务负载升至约 `3.0` 左右时，低价会通过更高委托率把系统推入 backlog 通道。
3. 任务负载是本组实验里最强的压力驱动项，价格摩擦更多像一个放大器或缓冲器，而不是主导变量。
4. 第一组实验已经足够支持一个稳健判断：便利并不会凭空消灭劳动，它更常见的作用是改变劳动在系统中的分布，并在容量边界附近放大积压。

## 3. Package A：日常摩擦不是孤立事件，而是时间分配结构的外显

`Package A` 比较了 Type A 与 Type B 在不同 horizon 下的尾窗表现。关键结果如下：

| 步数 | Type A 压力 | Type B 压力 | Type B 相对 Type A | Type A 总劳动 | Type B 总劳动 | 劳动差异 | Type A 委托占比 | Type B 委托占比 | Type A 最终可支配时间 | Type B 最终可支配时间 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: | ---: |
| 120 | 0.0373 | 0.0534 | +43.2% | 432.18 | 567.07 | +31.2% | 0.0965 | 0.6480 | 3.7672 | 2.3667 |
| 200 | 0.0413 | 0.0492 | +19.3% | 436.27 | 565.18 | +29.5% | 0.0947 | 0.6465 | 3.5183 | 2.1961 |
| 300 | 0.0374 | 0.0476 | +27.2% | 433.51 | 563.27 | +29.9% | 0.0891 | 0.6444 | 3.6432 | 2.4443 |

这组结果有三个值得强调的含义：

- 差异不是短期噪声。无论 horizon 取 120、200 还是 300 步，Type B 的总劳动始终比 Type A 高约 30%。
- Type B 的“便利”是以更低的平均可支配时间换来的。三组 horizon 下，Type B 的最终平均可支配时间都比 Type A 低约 1.2 到 1.4 小时。
- Type B 的高委托结构并没有把系统带向更低压力，反而把压力维持在更高平台。

就 H1 而言，这一部分已经给出强支持：更高委托率与更高总劳动小时数是稳定同向关系。

## 4. Package C：价格摩擦只能解释一部分，真正的转折点在容量阈值

### 4.1 上下文扫描：低价服务并不是总是“坏”，但它只在低负载区有效

`service_cost_context_scan` 比较了四类上下文下的最低价与最高价情景：

| 上下文 | 低价压力 | 高价压力 | 低价总劳动 | 高价总劳动 | 低价委托占比 | 高价委托占比 |
| --- | ---: | ---: | ---: | ---: | ---: | ---: |
| Default | 0.0204 | 0.0379 | 501.71 | 494.88 | 0.4624 | 0.2397 |
| Type A | 0.0228 | 0.0556 | 439.07 | 435.81 | 0.2114 | 0.0526 |
| Type B | 0.0505 | 0.0572 | 566.15 | 555.41 | 0.6835 | 0.4534 |
| Overloaded | 1.0000 | 1.0000 | 800.00 | 797.08 | 0.9974 | 0.9711 |

这张表可以直接读出两点：

- 在 `Default`、`Type A`、`Type B` 三种未过载上下文中，低价确实会减压，但同时也会提高总劳动和委托占比。
- 一旦进入过载区间，压力会同时饱和到 `1.0`，此时价格的主要作用不再是“降低压力”，而是改变积压规模。

`research_metric_probe.csv` 把这一点说得更清楚：

- `default` 情景下，低价与高价都没有 backlog，`match_rate = 1.0`
- `overloaded` 情景下，低价 backlog 为 `47559.71`，高价 backlog 为 `11693.83`，低价约为高价的 `4.07` 倍
- 在 `type_b` 语境下，低价 backlog 已经开始出现，而高价几乎没有 backlog

因此，价格摩擦不是一个可以脱离任务负载单独解读的变量。它必须与系统是否接近供给上限一起看。

### 4.2 阈值扫描：大约在 `tasks_per_step_mean = 3.0` 左右发生符号翻转

`service_cost_task_load_atlas` 是本组实验最关键的证据。它只看两档更接近阈值的委托水平：

- 中等委托：`delegation_preference_mean = 0.55`
- 高委托：`delegation_preference_mean = 0.72`

代表性结果如下：

| 委托均值 | 任务负载 | 低价压力 | 高价压力 | 低价 backlog | 高价 backlog |
| --- | ---: | ---: | ---: | ---: | ---: |
| 0.55 | 2.50 | 0.0160 | 0.0315 | 0.0000 | 0.0000 |
| 0.55 | 3.00 | 0.2826 | 0.2044 | 0.3500 | 0.0000 |
| 0.55 | 3.50 | 1.0000 | 1.0000 | 86.3000 | 44.7250 |
| 0.72 | 2.50 | 0.0135 | 0.0203 | 0.0000 | 0.0000 |
| 0.72 | 3.00 | 0.4620 | 0.2130 | 1.3250 | 0.2250 |
| 0.72 | 3.50 | 1.0000 | 1.0000 | 17899.3875 | 100.7875 |

这说明：

- 在 `2.5` 负载附近，低价服务仍然是减压阀。
- 到 `3.0` 左右，低价开始变成放大器，尤其在高委托情景下更明显。
- 到 `3.5`，系统整体已进入饱和区，低价与高价在平均压力上都显示为崩溃，但低价会制造远大于高价的 backlog。

### 4.3 机制解释：低价不是直接抬高压力，而是先抬高委托，再抬高积压

`research_metric_probe.csv` 还能补出机制链条：

- 中等委托、负载 `3.0` 时，低价相对高价的 `match_rate` 从 `1.0000` 降到 `0.9981`，`stress_breach_share` 从 `0.5821` 升到 `0.6551`
- 高委托、负载 `3.0` 时，低价相对高价的 `match_rate` 从 `0.9987` 降到 `0.9949`，`stress_breach_share` 从 `0.6191` 升到 `0.7146`

也就是说，价格并不是直接进入 stress 方程，而是通过以下链路起作用：

1. 低价提高已实现委托比例
2. 更多任务进入服务池
3. provider capacity 开始吃紧后，匹配失败增加
4. 失败任务回流 backlog
5. backlog 持续侵蚀后续时间预算，导致更多 agent 穿越 stress threshold

### 4.4 机制拆解：任务负载是主变量，价格是放大器

`preset_decomposition_v2` 的作用，是把 Type A 与 Type B 的预设拆成单因素变化。

对 Type A 基线而言：

- 把任务负载均值从 `2.2` 提到 `2.8`，`tail_avg_stress_mean` 上升 `+0.1045`，`tail_total_labor_hours_mean` 上升 `+106.99`
- 仅把 `service_cost_factor` 从 `0.65` 降到 `0.20`，压力下降 `-0.0168`，但委托占比上升 `+0.0876`

对 Type B 基线而言：

- 把任务负载均值从 `2.8` 降回 `2.2`，压力下降 `-0.0439`，总劳动下降 `-117.35`
- 仅把 `service_cost_factor` 从 `0.20` 提回 `0.65`，压力只上升 `+0.0040`

结论很明确：  
真正决定系统是否跨过阈值的，是任务压力；价格摩擦主要决定系统跨阈值后是“温和过载”还是“快速积压”。

## 5. 对四个研究假设的含义

### H1：更高委托率会提高系统总劳动小时数

本组实验支持 H1。  
证据来自两部分：

- `Package A` 中 Type B 的总劳动始终比 Type A 高约 30%
- `Package C` 中低价通常提高委托占比，同时使总劳动略高于高价情景

### H2：存在触发 involution spiral 的临界阈值

本组实验支持 H2。  
当前最稳健的阈值判断是：在研究引擎参数下，`tasks_per_step_mean` 大约到 `3.0` 左右时，低价服务会从减压阀翻转为积压放大器。

### H3：更高自治会降低便利感，但提升总体福祉

本组实验对 H3 给出部分支持。  
“便利感”没有直接度量，但更高自治确实对应：

- 更低总劳动
- 更低平均压力
- 更高剩余可支配时间

因此，如果把福祉近似理解为“较低压力 + 较高可支配时间”，当前证据更偏向支持自治型配置。

### H4：中间系统不稳定，会向极端漂移

本组实验没有直接检验 H4。  
这一组更偏向价格阈值与容量机制，不足以就 mixed-system stability 做出结论。

## 6. 局限与下一步

这组实验已经足够支撑“价格作用具有阈值条件”这一结论，但仍有三个边界需要保留：

- 价格是外生变量，因此不能用这组结果去声称现实世界的价格形成机制或价格内生反馈。
- `delegation_labor_delta` 在 backlog 爆炸时会变得难以解释，不能把它直接当成福利汇总指标。
- `Package C` 在本次 campaign 中没有登记图像资产，研究报告主要依赖 `combo_summary.csv` 与 `research_metric_probe.csv` 的数值证据。

如果下一轮要继续推进，优先建议：

1. 为 `service_cost_task_load_atlas` 增加更密的 `2.75-3.25` 负载剖面
2. 用 backlog-adjusted outstanding-work 指标替代当前的 `delegation_labor_delta`
3. 把本组得到的容量阈值，和更大规模实验中的 delegation-threshold 结果拼接起来做统一解释
